## 问题诊断
- 判定来源：后端通过 `DetectionRun.Summary.HasIssue` 更新图片状态，路径 `internal/models/detection.go:129-156`，`hasIssue=true → status=“缺陷”`；否则为“合格”，未检测时为“未检测”。
- 触发更新：所有写入检测结果的入口都会调用 `InsertDetectionRun()`，其内部调用 `UpdateImageDetectionSummary()` 更新图片表；参考 `internal/models/detection.go:100-126`。
- 现象原因：
  - 外部推理服务有时未返回 `summary`（或返回空值），而 `handlers/detections.go` 在 `CreateImageDetection` 未进行兜底计算，导致 `hasIssue=false`，图片被更新为“合格”或保持“未检测”。代码参考 `internal/handlers/detections.go:152-175, 177-195`。
  - 即使图片摘要更新失败（例如 ID 不匹配或数据库错误），`InsertDetectionRun` 忽略了 `UpdateImageDetectionSummary` 的错误返回，导致图片状态不变而无日志/反馈。

## 修复方案
- 补充缺陷汇总兜底逻辑：当请求体未提供 `summary` 或为默认空值时，依据 `items` 自动生成：
  - `hasIssue = len(items) > 0`
  - `issueType = "auto"`（或第一项类别名）
  - `objectCount = len(items)`
  - `avgScore = items.confidence 的均值`
- 服务路径兜底：在 `StartImageDetect` 的外部服务调用分支，若 `dr.Summary` 为空但 `dr.Items` 非空，也按同样规则兜底生成 `summary`。
- 严格错误传播：`InsertDetectionRun` 中对 `UpdateImageDetectionSummary` 的错误不再忽略；若更新失败，直接返回错误，让调用方记录失败并反馈。

## 具体改动
- 文件 `internal/handlers/detections.go`
  - 在 `CreateImageDetection()` 写入前，检测 `req.Summary` 是否为空（`IssueType==""` 且 `ObjectCount==0` 且 `AvgScore==0`）；若为空且 `req.Items` 非空，计算并填充 `req.Summary`。
  - 计算均值的本地小函数（或内联），避免依赖服务层私有函数。
- 文件 `internal/services/detect_job.go`
  - 在外部服务分支解析后（`json.NewDecoder(resp.Body).Decode(&dr)` 之后），若 `dr.Summary` 为空但 `dr.Items` 非空，则计算兜底 `summary`；随后再写库。
- 文件 `internal/models/detection.go`
  - 在 `InsertDetectionRun()` 的 3 处调用 `UpdateImageDetectionSummary()`，改为检查错误并向上返回，确保调用方感知更新失败。

## 验证用例
- 用例1：调用 `POST /images/{id}/detections`，`items` 非空但 `summary` 为空 → 期望图片 `status="缺陷"`，`isDetected=true`。
- 用例2：同上但 `items` 为空 → 期望 `status="合格"`。
- 用例3：通过 `POST /images/{id}/detect`（服务分支返回 `items` 非空但 `summary` 缺失）→ 期望兜底生效，图片 `status="缺陷"`。
- 用例4：模拟数据库更新失败（例如传入不存在的 `imageID`）→ 期望 API 返回 500 并记录错误，不出现“未检测”静默失败。

## 兼容性与注意
- 不改动现有路由与请求结构；仅在后端内部增加兜底与错误传播，符合 Gin 与 swaggo 注解约定。
- 不增加外部依赖；只编辑现有文件。
- 若后续需更精细判定（如仅特定类别视为缺陷），可在配置中增加缺陷类别白名单，但本次先以兜底修复状态错判为目标。
